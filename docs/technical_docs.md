# Technical Documentation

## 1. System Architecture Overview

The `dpadmin` application is a monolithic web application built using the Laravel framework. The application follows the traditional MVC (Model-View-Controller) architecture, rendering server-side views and managing API/backend operations simultaneously. It heavily relies on the Laravel routing system, Eloquent ORM for database interactions, and Blade templates for the frontend.

The project is structured within the `public_html` directory to conform to legacy cPanel/shared hosting deployments, where the document root is the application structure itself. In a modern/Dockerized setup, the document root should strictly point to `public_html/public`.

## 2. Technology Stack

- **Backend:** PHP (Framework: Laravel)
- **Database:** MySQL / MariaDB
- **Frontend:** HTML, CSS, JavaScript (Bootstrap, jQuery, Vue/React if present in `resources/js`), compiled via Webpack Mix.
- **Web Server:** Nginx or Apache (currently structured with `.htaccess` for Apache).
- **Environment Management:** dotenv (`.env`) for configuration.
- **Dependency Management:** Composer for PHP packages, NPM/Yarn for frontend assets (`package.json`).

## 3. Directory Structure (`public_html`)

- `app/`: Contains the core PHP code for the application.
  - `Http/Controllers`: Contains the application controllers (Admin logic, API, Base Controllers).
  - `Model/`: Contains the Eloquent ORM models representing database tables (`Customer`, `OrderCustomer`, etc.).
  - `MrData.php`: A core utility class for text processing, data filtering, and formatting.
- `bootstrap/`: Contains the framework bootstrapping script and cache configuration.
- `config/`: Contains all configuration files (database, mail, session, app, etc.).
- `database/`: Contains database migrations and seeds.
- `public/`: The HTTP document root containing `index.php`, CSS, JS, and image assets. (Contains `admin` and `css`/`js` compilation outputs).
- `resources/`: Contains raw assets, language files, and Blade templates (`views`).
- `routes/`: Contains all route definitions.
  - `web.php`: Defines all web-accessible endpoints.
  - `api.php`: Defines API endpoints (if any).
- `storage/`: Contains compiled Blade templates, file sessions, file caches, and other files generated by the framework.
- `vendor/`: Composer dependencies (ignored by Git, managed via `composer install`).

## 4. Application Setup & Configuration Guide

### Current (Legacy) Deployment:
Currently deployed directly to a server path (`cwp` or similar shared hosting), relying on PHP-FPM/CGI and Apache `.htaccess`.

### Local Development Setup:
1. Clone the repository.
2. Navigate to `public_html`.
3. Copy `.env.example` to `.env` and fill in DB credentials and Application Key.
4. Run `composer install` to verify PHP dependencies.
5. Run `npm install` and `npm run dev` to compile frontend assets.
6. Run `php artisan migrate` to execute database schema creation.
7. Run `php artisan serve` to start a local PHP development server.

## 5. Security & Authentication

- The application uses Laravel's built-in authentication mechanisms via `Auth\AuthController`.
- Routes grouped under `admin/` use the `['middleware' => ['auth']]` array, preventing unauthorized access.
- There is a OneDrive OAuth integration flow found in `OnedriveController`.
